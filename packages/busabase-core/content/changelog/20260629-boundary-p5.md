## What Changed

Two confirmed P0 fixes (both covered by `tests/boundary-p5.test.ts`):

1. **CR merge guard against archived target records** (`logic/cr-lifecycle.ts`)
   - The merge engine validated views (`status !== "active"`) but **not** records.
     A change request authored while a record was active could merge after the
     record was archived by another CR, silently mutating an archived record.
   - Added a per-operation lifecycle guard mirroring the view check:
     `record_update` / `record_delete` require an **active** target;
     `record_restore` requires an **archived** target. Throws `CONFLICT` otherwise.
   - Also scoped the target-record load to the CR's own `baseId`, so a crafted CR
     cannot reach a record in another base/space via a leaked id.

2. **Space-scoped comment subject resolution** (`logic/audit.ts`)
   - `resolveCommentSubject` looked up the record / change-request / operation /
     commit **by id only**, with no `spaceId` filter. A user in space A could
     create a comment linked to an entity in space B (cross-tenant leak).
   - All four branches now filter by `getContextSpaceId()`.

## Why

Both are data-integrity / multi-tenant isolation gaps surfaced by a boundary audit
that extended the P4 archive/restore + spaceId work to the cross-entity surface
(change requests, comments).
